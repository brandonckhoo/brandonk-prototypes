Conductor · Feature proposal

Make settings and code clickable in Conductor's answers

Click a setting Conductor mentions and it takes you there and highlights it.

The problem

Right now Conductor's answers are one grey wall of text. Setting names and code are mixed into the sentences, so you can't tell them apart, and you can't click them. You read the name of a setting, then go digging through the builder to find it yourself.

Today
"I raised the interruption sensitivity to 0.9 and set response_eagerness to 1 with dynamically_adjust off, so the agent responds instantly after the caller stops."

Flat text. Nothing stands out, nothing is clickable.

Proposed
The agent responds the instant the caller stops. That's ⚙ Response Eagerness at its max.
config⧉ Copy
response_eagerness: 1
dynamically_adjust: off

Click the chip, you land on that setting. Code has its own block you can copy.

One mechanism, two views

Both views run the same engine underneath: find the exact spot, switch to the right surface, highlight it, focus the field. They differ only by what the user is doing. When Conductor is explaining, it drops a chip. When it needs you to act, it hands you a checklist.

Inline chip — for referencing
Conductor names a setting, node, or variable in its answer. Click the chip and it jumps you there and highlights it. Woven into the sentence, so you read on or go look.
Action card — for setup tasks
When something needs your input before the agent can run, Conductor lists it as a checklist with buttons like Connect Cal.com or Go to node. A task list, not a sentence.

Code and config keep their own copyable block, set apart from the prose. We dropped the earlier "What's set now" summary list, it only repeated the chips already in the sentence.
