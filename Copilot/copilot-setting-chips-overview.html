Conductor · Feature proposal

Setting chips & code blocks

Conductor should render the settings and code it references as distinct objects, not plain text buried in prose. Settings become clickable chips that jump to and highlight the matching field. Code renders in copyable code blocks. The same idea Cursor uses when it links a file and line range back to the editor.

The problem

Conductor outputs settings and code as plain text mixed into prose, so it is hard to tell where a setting name ends and the sentence continues. And when it names a setting, the user has to hunt through the builder to find it. Let the user click a reference in the response to open and highlight the matching content.Conductor v2 feedback

How it works

1

Conductor tags references

As it writes its answer, Conductor knows the exact setting or node it names. It attaches that target to the chip, and wraps any config it emits in a code block.

2

The user clicks a chip

Clicking opens the section, scrolls to the field, and pulses a highlight ring. When no field can be resolved, the chip falls back to plain inline code so the reference is never a dead link.

3

References resolve anywhere

A chip points to a settings field, a global speech setting, or a node on the canvas. Conductor switches to the right surface and stays docked, so the user never loses the conversation.

Where a reference resolves

Settings field agent-wide
  • Speech timing — response_eagerness, interruption_sensitivity
  • Model and voice selection
  • Webhook URL, reminder frequency
  • Post-Call Analysis fields
  • Security & fallback settings
Node on the canvas per node
  • A node's prompt or transition condition
  • Function parameters — event IDs, headers
  • Transfer number on a Transfer node
  • Knowledge base attached to a node
  • The node itself, panned and highlighted
